# Token 效率规范 (Token Efficiency Standards)

减少不必要的 token 消耗，同时保证代码质量和正确性。

## 核心原则

**复用优先，生成兜底。** 已存在的、经过验证的代码是资产，不是草稿。

## 迁移与复制

### 同项目/跨项目文件迁移

当目标文件在老项目或同项目其他位置已存在时：

1. **禁止重新生成整个文件** — 先复制原文件，再做定向小改（import 路径、配置读取方式等）。
2. **复制命令优先** — 使用 `copy` / `cp` 命令复制文件，而非用 `write_to_file` 重新输出全部内容。
3. **改动必须最小化** — 只改确实需要适配新项目结构的部分（通常是 import 路径、日志初始化、配置读取）。
4. **禁止"顺便精简"** — 不得在迁移过程中删减功能代码（如删除已验证的映射表条目、去掉已有的边界处理）。精简是独立的重构任务，不与迁移混合。

### 判定规则

| 场景 | 正确做法 | 错误做法 |
|------|----------|----------|
| 老项目有 500 行文件，新项目需要 | copy + 改 5 行 import | 重新生成 300 行"精简版" |
| 老项目有函数，新项目需要相同逻辑 | 复制函数 + 改调用签名 | 从头写同样的函数 |
| 需要部分功能 | 复制全文件 + 后续独立 PR 删减 | 重新写一份只含部分功能的版本 |

## 编辑策略

### 小改用 edit，大改才用 write

- **改动 < 30 行**：使用 `edit` / `multi_edit` 工具做定向替换。
- **改动 30–100 行**：拆分为多个 `edit` 调用，每次改一个逻辑块。
- **改动 > 100 行或全新文件**：才使用 `write_to_file`。

### 避免重复输出

- **不重复读取已读过的文件** — 在同一会话中，如果已经读过某文件且内容未改变，直接引用行号而非再次读取。
- **不输出未改动的代码** — 向用户解释时，只引用文件路径 + 行号，不复制粘贴大段未改动的代码。
- **批量独立编辑并行执行** — 多个不相关文件的编辑用并行 tool call，不逐个串行。

## 搜索策略

### 精准搜索优于广撒网

- 先用 `code_search` 做语义定位，再用 `grep_search` 精确匹配。
- `grep_search` 默认不开 `MatchPerLine`，只在确认结果少时才开。
- 搜索范围尽量缩小到具体子目录，避免全仓库扫描。

## 回复策略

### 简洁优先

- 代码变更直接用工具执行，不在聊天中输出代码块（除非用户明确要求看代码）。
- 总结用表格或短列表，不用长段落叙述。
- 引用已有代码用 `@filepath:line` 格式，不复制内容。

## 会话上下文管理

### 利用已有上下文

- 会话中已读过的文件内容、已执行的搜索结果，在后续步骤中直接复用，不重复获取。
- 如果 checkpoint 或 summary 已提供了文件内容摘要，优先使用摘要而非重新读取文件。
