# 文档归类规范 (Doc Standards)

所有在 `docx/` 目录下创建、移动、重命名的 Markdown 文档必须遵循以下规则。

## 文件命名

- 格式：`YYYYMMDD_文档名称.md`
- **YYYYMMDD** 为文档创建日期（非修改日期）
- 名称使用简短中文，词间用下划线 `_` 连接
- 统一 `.md` 格式，禁止 `.txt`、`.docx` 等

## 目录分类

| 目录             | 用途           | 典型内容                                     |
| ---------------- | -------------- | -------------------------------------------- |
| `01_项目管理/` | 项目级管理文档 | 开发计划、里程碑、排期、启动指南、部署流程   |
| `02_技术文档/` | 技术设计与实现 | 架构设计、API 规格、模块设计、外部服务集成   |
| `03_设计规范/` | 设计标准与规范 | UI 设计规范、代码规范、画风规范、Prompt 模板 |
| `04_测试报告/` | 测试与实验     | 版本测试总结、性能测试、实验记录             |
| `05_工作进度/` | 会话级工作记录 | 每次开发会话的完成事项、变更摘要、状态快照   |

### 分类判定规则

1. **工作进度**（`05_`）：记录某次开发会话中做了什么、改了什么、当前状态。侧重"过程记录"。
2. **项目管理**（`01_`）：面向项目整体的规划、指南类文档。侧重"计划与流程"。
3. **技术文档**（`02_`）：描述某个模块/系统的技术设计。侧重"怎么做"。子目录按领域划分：
   - `前端/` — Vue、CSS、组件、路由等前端相关
   - `后端/` — FastAPI、Python 服务、API 等后端相关
   - `通用/` — 跨前后端的 Pipeline、外部服务、工具等
4. **设计规范**（`03_`）：约束性的标准文档，全项目统一遵守。侧重"规则与约束"。
5. **测试报告**（`04_`）：测试结论和实验数据。侧重"验证结果"。

### 易混淆场景

| 场景                           | 归类                  | 原因           |
| ------------------------------ | --------------------- | -------------- |
| "今天完成了 TTS 组件重构"      | `05_工作进度/`      | 会话级工作记录 |
| "TTS 模块的技术架构与接口设计" | `02_技术文档/后端/` | 技术方案文档   |
| "所有组件的代码风格要求"       | `03_设计规范/`      | 约束性规范     |
| "启动脚本使用说明"             | `01_项目管理/`      | 项目级操作指南 |
| "TTS 合成质量对比实验"         | `04_测试报告/`      | 实验验证       |

## README 索引

- 每次新增/移动/删除文档后，必须同步更新 `docx/README.md` 的文档索引表。
- 索引按目录分节，每条包含文件名和一句话说明。

## 02_技术文档子目录

`02_技术文档/` 下按领域分子目录：

- `前端/` — 前端架构、组件、路由、UI 相关
- `后端/` — 后端架构、API、服务、数据库相关
- `通用/` — Pipeline、外部 API、工具链等跨端内容

新建技术文档时必须放入对应子目录，禁止直接放在 `02_技术文档/` 根目录。
