# 测试驱动开发

## 核心理念

**基本原则**：测试应通过公开接口验证行为，而非实现细节。代码可以完全重写；测试不应因此失败。

**好的测试**是集成式的：通过公开 API 走真实代码路径。描述系统"做什么"而非"怎么做"。

**坏的测试**耦合了实现：mock 内部协作者、测试私有方法、或通过外部手段验证。

## 反模式：水平切片

**不要先写完所有测试再写所有实现。** 这是"水平切片"。

**正确方式**：通过示踪弹进行垂直切片。一个测试 → 一个实现 → 重复。

```
错误（水平）：
  红灯：test1, test2, test3, test4, test5
  绿灯：impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  红→绿：test1→impl1
  红→绿：test2→impl2
  红→绿：test3→impl3
  ...
```

## 工作流程

### 1. 规划

写代码之前：

- [ ] 与用户确认需要哪些接口变更
- [ ] 与用户确认要测试哪些行为（按优先级排序）
- [ ] 寻找深模块机会（小接口，深实现）
- [ ] 为可测试性设计接口
- [ ] 列出要测试的行为（不是实现步骤）
- [ ] 获得用户批准

问："公开接口应该长什么样？哪些行为最重要需要测试？"

### 2. 示踪弹

写一个测试来确认系统的一个行为：

```
红灯：写第一个行为的测试 → 测试失败
绿灯：写最少的代码让它通过 → 测试通过
```

### 3. 增量循环

对每个剩余行为：

```
红灯：写下一个测试 → 失败
绿灯：写最少的代码 → 通过
```

规则：
- 一次一个测试
- 只写刚好让当前测试通过的代码
- 不要提前考虑后续测试
- 测试聚焦于可观察的行为

### 4. 重构

所有测试通过后，寻找重构机会：

- [ ] 提取重复代码
- [ ] 深化模块（把复杂性藏在简单接口后面）
- [ ] 在自然的地方应用 SOLID 原则
- [ ] 思考新代码揭示了现有代码的什么问题
- [ ] 每次重构后都跑测试

**红灯状态下绝不重构。** 先到绿灯再说。

## 每轮检查清单

```
[ ] 测试描述的是行为，不是实现
[ ] 测试只用公开接口
[ ] 内部重构不会导致测试失败
[ ] 代码是为当前测试写的最少量
[ ] 没有添加投机性功能
```
