# 架构优化

挖掘架构摩擦点，提出**深化机会** — 让浅模块变深的重构。

## 术语表

严格使用以下术语：

- **模块** — 任何有接口和实现的东西（函数、类、包、切片）
- **接口** — 调用方使用模块所需了解的一切：类型、约束、错误模式、顺序、配置。不只是类型签名
- **实现** — 内部代码
- **深度** — 接口处的杠杆率。**深** = 高杠杆。**浅** = 接口几乎和实现一样复杂
- **接缝** — 接口所在之处；不需要原地编辑就能改变行为的位置
- **适配器** — 在接缝处满足接口的具体实现
- **杠杆** — 调用方从深度中获得的好处
- **局部性** — 维护者从深度中获得的好处：改动、bug、知识集中在一处

关键原则：

- **删除测试**：想象删掉这个模块。如果复杂性消失了，它就是一个透传。如果复杂性重新出现在 N 个调用方中，它就值得存在。
- **接口就是测试面。**
- **一个适配器 = 假设性接缝。两个适配器 = 真实接缝。**

## 流程

### 1. 探索

先读项目的领域术语表和相关 ADR。然后有机地遍历代码库，记录摩擦点：

- 哪里理解一个概念需要在多个小模块间跳转？
- 哪里的模块是**浅**的？
- 哪里为了可测试性提取了纯函数，但真正的 bug 藏在调用方式中（没有**局部性**）？
- 哪里的紧耦合模块在接缝处泄漏？
- 哪些部分没有测试，或通过当前接口很难测试？

对任何可疑的东西应用**删除测试**。

### 2. 以 HTML 报告呈现候选项/或者MD文档（默认选择MD文档总结）

在系统临时目录（Windows 上是 `%TEMP%`）写一个自包含 HTML 文件。用 Tailwind CDN 做样式，Mermaid CDN 画图。

每个候选项包含：

- **涉及文件** — 哪些文件/模块
- **问题** — 为什么当前架构造成摩擦
- **方案** — 用白话描述会改什么
- **收益** — 用局部性和杠杆来解释
- **前后对比图** — 并排展示
- **推荐强度** — `强烈推荐`、`值得探索`、`试探性`

末尾加一个**首选推荐**小节。

先不要提议具体接口。问用户："你想深入探索哪个？"

### 3. 烤问循环

用户选了候选项后，进入烤问对话。遍历设计树 — 约束、依赖、深化后的模块形状、接缝背后是什么、哪些测试能存活。

决策确定时即时产生副作用：

- 给新概念起名？加到 `CONTEXT.md`。
- 锐化模糊术语？更新 `CONTEXT.md`。
- 用户以重要理由拒绝候选项？提议写 ADR。
