# 诊断调试

专治疑难 bug 的纪律流程。跳过任何阶段必须明确说明理由。

## 阶段 1 — 建立反馈循环

**这是核心技能。** 其余都是机械操作。如果你有一个快速、确定性的、可由 agent 运行的通过/失败信号，你就能找到根因。

在这里花不成比例的精力。**要激进、要有创造力、绝不放弃。**

### 构建方式 — 按优先级尝试

1. **失败测试** — 在能触达 bug 的接缝写单元/集成/端到端测试
2. **Curl / HTTP 脚本** — 对开发服务器发请求
3. **CLI 调用** — 用固定输入，diff stdout 与已知正确快照
4. **无头浏览器脚本**（Playwright / Puppeteer）— 驱动 UI，断言 DOM/控制台/网络
5. **回放捕获的 trace** — 把真实请求/载荷/事件日志存盘，隔离回放
6. **一次性测试夹具** — 启动系统最小子集来走 bug 路径
7. **属性/模糊测试循环** — 如果 bug 是"偶尔输出错误"，跑 1000 次随机输入
8. **二分夹具** — 自动化"在状态 X 启动 → 检查 → 重复"，配合 `git bisect run`
9. **差异循环** — 同一输入走旧版和新版，diff 输出
10. **人工辅助脚本** — 最后手段，需要人点击时使用

### 迭代循环本身

- 能不能更快？
- 信号能不能更清晰？
- 能不能更确定性？

### 确实无法建立循环时

明确说出来。列出你尝试了什么。向用户请求：(a) 能复现的环境访问权限，(b) 已捕获的制品（HAR 文件、日志转储、截屏），(c) 添加临时生产插桩的许可。**没有循环不要进入假设阶段。**

## 阶段 2 — 复现

运行循环。观察 bug 出现。确认：

- [ ] 循环产生的是**用户描述的**故障模式
- [ ] 多次运行可复现
- [ ] 已记录精确症状（报错信息、错误输出、耗时）

## 阶段 3 — 假设

在测试任何假设之前，先生成 **3-5 个排序假设**。

每个假设必须**可证伪**：说明它做出的预测。

> 格式："如果原因是 <X>，那么 <改变 Y> 会让 bug 消失 / <改变 Z> 会让它更严重。"

**测试前先把排序列表展示给用户。**

## 阶段 4 — 插桩

每个探针必须对应阶段 3 的一个具体预测。**每次只改一个变量。**

工具偏好：
1. **调试器 / REPL 检查** — 如果环境支持
2. **针对性日志** — 放在能区分假设的边界处
3. 绝不"到处打日志然后 grep"

**给每条调试日志打唯一前缀标签**，如 `[DEBUG-a4f2]`。清理时一条 grep 搞定。

## 阶段 5 — 修复 + 回归测试

**先写回归测试再修复** — 但仅在存在正确接缝时。

1. 把最小化复现转为该接缝的失败测试
2. 看它失败
3. 应用修复
4. 看它通过
5. 用阶段 1 的循环重新验证原始（未最小化的）场景

## 阶段 6 — 清理 + 复盘

- [ ] 原始复现不再出现
- [ ] 回归测试通过
- [ ] 所有 `[DEBUG-...]` 插桩已移除
- [ ] 一次性原型已删除
- [ ] 正确假设写入了 commit/PR 信息

**然后问：什么能预防这个 bug？** 如果答案涉及架构变更，交给 `/架构优化` 处理。
