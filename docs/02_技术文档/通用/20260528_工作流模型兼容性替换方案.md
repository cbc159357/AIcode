## 零、RunningHub 官方模型列表 API（调查结论）

> **结论：RunningHub 提供官方模型列表 API，且包含架构族字段，足以支撑兼容性方案。**

### API 端点

```
POST https://www.runninghub.cn/openapi/v2/resource/list
Authorization: Bearer {API_KEY}
Content-Type: application/json

请求体: {} （空，支持分页参数待验证）
```

### 响应结构（完整字段）

```json
{
  "code": 0,
  "data": {
    "records": [
      {
        "id": "2039188122948345857",
        "resourceName": "realDream_Klein",
        "resourceType": "UNET",
        "nodeModelName": "realDream_klein9BV1.safetensors",
        "createTime": "2026-04-01 11:48:28",
        "desc": "<p>搬运</p>",
        "posterUrl": "https://...",
        "owner": { "name": "张某" },
        "tags": [
          { "id": 11, "name": "摄影" },
          { "id": 16, "name": "写实" }
        ],
        "versions": [
          {
            "version": "V1.0",
            "versionResourceName": "models/unet/realDream_klein9BV1.safetensors",
            "baseModel": "Flux2-Klein-9B",
            "baseModelSubtype": "",
            "triggerWords": "",
            "desc": ""
          }
        ]
      }
    ],
    "size": 10,
    "current": 1,
    "total": 1135,
    "pages": 114,
    "hasNext": true
  }
}
```

### 关键字段分析

| 字段 | 含义 | 用于兼容性方案 |
|------|------|--------------|
| `resourceType` | 节点类型（UNET / CLIP / VAE / LORA 等） | ✅ 知道这个模型用在哪种 Loader 节点 |
| `nodeModelName` | 文件名（如 `realDream_klein9BV1.safetensors`） | ✅ 与工作流 JSON 中 fieldValue **完全匹配** |
| `versions[].baseModel` | 架构族（如 `Flux2-Klein-9B` / `SD 1.5` / `SDXL`） | ✅ **直接判断兼容性** |
| `versions[].versionResourceName` | 完整路径（如 `models/unet/xxx.safetensors`） | ✅ 作为 fieldValue 传给 API |
| `versions[].triggerWords` | LoRA 触发词 | 📌 可展示给用户，自动注入提示词 |
| `tags` | 风格标签（摄影/写实/动漫等） | 📌 辅助模型筛选 |

### 重要发现

1. **有 baseModel 字段** — 直接告知架构族，无需正则推断文件名
2. **总量 1135 个**（2026-05-28），分 114 页，需要全量缓存或按需查询
3. **没有明确的量化字段**（int8/fp8/gguf 等量化信息可能在 `desc` 中或文件名中体现，如 `flux_dev_fp8_e4m3fn.safetensors`）
4. **nodeModelName 直接对应工作流 fieldValue** — 这是关键：工作流 JSON 里 `UNETLoader.unet_name` 的值就是 `nodeModelName`

### 量化信息获取

RunningHub API 暂**无单独量化字段**，量化信息从以下途径推断：

| 推断来源 | 示例 | 可信度 |
|---------|------|--------|
| 文件名关键词 | `fp8_e4m3fn` / `int8` / `gguf` / `nf4` | 高（约 80%） |
| `desc` 字段 HTML 内容 | 搬运说明可能提及量化 | 低 |
| `baseModelSubtype` 字段 | 目前为空，未来可能填充 | 未知 |

## 一、问题描述

### 背景

RunningHub 工作流在云端执行，用户在本地 UI 中修改参数（delta 覆盖云端默认值）。工作流中典型的"模型节点"包括：

| 节点类型 | 字段 | 示例值 |
|----------|------|--------|
| `UNETLoader` | `unet_name` | `flux_dev.safetensors` |
| `CLIPLoader` | `clip_name1` / `clip_name2` | `clip_l.safetensors`, `t5xxl_fp8_e4m3fn.safetensors` |
| `VAELoader` | `vae_name` | `ae.safetensors` |
| `LoraLoader` | `lora_name` | `anime_style_v2.safetensors` |
| `CheckpointLoader` | `ckpt_name` | `sdxl_base.safetensors` |

### 核心矛盾

```
用户修改: UNETLoader.unet_name = "sd_xl_base.safetensors"  (原为 Flux 模型)
          ↓
CLIP 节点: 仍指向 Flux 专用 CLIP (t5xxl + clip_l)  ← 不兼容!
VAE 节点:  仍指向 Flux VAE (ae.safetensors)        ← 部分可能兼容
LoRA 节点: 仍指向 Flux LoRA                         ← 不兼容!
          ↓
结果: RunningHub 运行时报错，用户不知道原因
```

### 用户诉求

- **不想**维护一张"模型 A 兼容模型 B、C、D"的硬编码对照表
- **希望**系统能智能感知兼容性，或以合理的方式引导用户做整体替换

## 二、问题根因分析

### 2.1 为什么会牵一发而动全身？

ComfyUI 架构中，模型兼容性由**架构族（Architecture Family）**决定：

```
架构族 → 决定 → UNET 结构 / 注意力头数 / 时间步编码方式
            ↓
       必须匹配 → CLIP 编码器类型（SD1.5 用 CLIP-L, SDXL 用 CLIP-G+L, Flux 用 T5+CLIP-L）
       必须匹配 → VAE 压缩率（SD1.5/SDXL 用 4x VAE, Flux/HiDream 用专用 AE）
       必须匹配 → LoRA 权重形状（不同架构的 attention 层维度不同）
```

| 架构族 | UNET | CLIP | VAE | LoRA |
|--------|------|------|-----|------|
| SD 1.5 | sd-v1-x.ckpt | clip-vit-l | vae-ft-mse | sd15 LoRA |
| SDXL | sdxl_base.ckpt | clip-g + clip-l | sdxl-vae | sdxl LoRA |
| Flux.1 | flux_dev.sft | t5xxl + clip-l | ae.sft | Flux LoRA |
| HiDream | hidream.ckpt | 4×CLIP | 专用 VAE | HiDream LoRA |

### 2.2 本地参数覆盖的局限性

用户只能修改**单个字段值**，但架构切换需要**原子性地修改多个节点的多个字段**。现有 delta 机制无法表达"一组关联字段联动修改"的语义。

## 三、方案探讨

### 方案 A：模型预设包（Model Preset Bundle）

#### 核心思想

将"需要联动替换的一组模型字段"打包成一个**命名预设**，用户选预设而不是逐个改模型。

```yaml
# 预设示例（存储在本地 / 后端）
preset_id: flux_dev_complete
name: "Flux.1-dev 完整套装"
architecture: flux1
nodes:
  - nodeId: "4"
    fieldName: unet_name
    fieldValue: flux_dev.safetensors
  - nodeId: "11"
    fieldName: clip_name1
    fieldValue: t5xxl_fp8_e4m3fn.safetensors
  - nodeId: "11"
    fieldName: clip_name2
    fieldValue: clip_l.safetensors
  - nodeId: "10"
    fieldName: vae_name
    fieldValue: ae.safetensors
```

#### 工作流

```
用户操作: 在"模型配置"分组中选择预设 "Flux.1-dev 完整套装"
         ↓
系统自动: 将预设中所有字段值注入 nodeInfoList 一起提交
         ↓
RunningHub: 使用完整且兼容的模型组合运行
```

#### 优点
- **用户体验最佳**：一次选择，不需要理解兼容性细节
- **完全无硬编码**：预设由用户/管理员定义，系统只做调度
- **可移植**：预设可以导出分享

#### 缺点
- 需要人工维护预设库（初始成本）
- 预设与具体工作流强绑定（节点 ID 耦合）

#### 适用场景
适合**相对固定的工作流**，用户在多套模型间切换（如 Flux vs SDXL）

---

### 方案 B：架构族自动推断 + 联动提示

#### 核心思想

从模型文件名中自动推断所属**架构族**，当用户修改某个模型时，系统检测到跨族切换，自动提示其他需要一起换的节点。

#### 实现逻辑

**Step 1：文件名 → 架构族推断**（后端，正则 + 关键词匹配）

```python
_ARCH_PATTERNS: list[tuple[re.Pattern, str]] = [
    (re.compile(r"flux", re.I),       "flux1"),
    (re.compile(r"sdxl|xl_base", re.I), "sdxl"),
    (re.compile(r"sd.?1[._-]?5|v1[._-]", re.I), "sd15"),
    (re.compile(r"hidream", re.I),     "hidream"),
    (re.compile(r"aura", re.I),        "aura"),
    (re.compile(r"t5xxl", re.I),       "flux1_clip"),  # CLIP 归属
]

def infer_arch(filename: str) -> str:
    for pattern, arch in _ARCH_PATTERNS:
        if pattern.search(filename):
            return arch
    return "unknown"
```

**Step 2：分组内模型架构一致性校验**（可在保存参数时触发）

```python
def validate_model_group(params: list[dict], raw_nodes: dict) -> list[dict]:
    """返回兼容性警告列表"""
    model_nodes = [p for p in params if p["type"] == "select" and "name" in p["fieldName"]]
    archs = {p["nodeId"]: infer_arch(str(p["currentValue"])) for p in model_nodes}
    # 找出架构不一致的节点对
    ...
```

**Step 3：前端警告 UI**

```
⚠️ 检测到架构不一致
   UNET: flux1 (flux_dev.safetensors)
   CLIP: sd15  (clip-vit-l.safetensors) ← 不兼容
   
   建议一起更换:
   [CLIPLoader #11] clip_name1 → t5xxl_fp8_e4m3fn.safetensors
   [CLIPLoader #11] clip_name2 → clip_l.safetensors
   
   [一键应用建议] [忽略]
```

#### 优点
- **无需预先维护预设**：基于文件名自动推断
- **非强制**：保留用户自由度，只是提示不阻断
- **可扩展**：增加新架构只需补充正则规则

#### 缺点
- 文件名不规范时推断失败（用户自定义命名的模型）
- 无法知道 RunningHub 上有哪些可用模型文件（只知道当前工作流的值）
- 只能检测已修改的参数，无法主动推荐替代值

#### 适用场景
适合**探索性使用**，用户知道自己在做什么，只需要兼容性预警

### 方案 C：依赖图分析 → 自动模型分组

#### 核心思想

分析工作流节点连线（`rawNodes` 中的 `inputs` 字段），找出哪些模型节点**共同服务于同一个 KSampler/Denoise 节点**，将它们自动归为一组，只允许组内整体替换。

#### 分析过程

```
UNETLoader(4) ──MODEL──→ KSampler(3)
CLIPLoader(11)──CLIP───→ CLIPTextEncode ──CONDITIONING──→ KSampler(3)
VAELoader(10) ──VAE────→ VAEDecode ─────────────────────→ (输出)
LoraLoader(5) ──MODEL──→ UNETLoader(4)  [修改 UNET 之前]

解析结果: {4, 11, 10, 5} 共同服务 KSampler(3) → 归为 "主模型组"
```

#### 实现思路

```python

    """
    沿着 MODEL/CLIP/VAE 边向下追踪到 KSampler，
    将同一 KSampler 的所有上游模型节点归为一组。
    """
    # 已有 BFS 基础设施 (_build_downstream_map / _infer_role_bfs)，可复用
    ...
    return [
        {"groupName": "主采样模型组", "samplerNodeId": "3",
         "modelNodeIds": ["4", "11", "10", "5"]}
    ]
```

前端 UI：将同组模型节点渲染为一个**联动编辑器**，修改其中任一触发整组提示。

#### 优点
- **完全基于工作流结构**，无需任何外部数据
- 可复用已有 BFS 推断引擎
- 分组粒度精准（多 KSampler 的工作流可识别多个独立模型组）

#### 缺点
- 只能识别"谁归一组"，不能推荐"换成什么"
- 需配合方案 A（预设）或方案 B（架构推断）才能形成完整替换体验

### 方案 D：RunningHub 官方模型元数据 API ⭐ 推荐

#### 核心思想（调查后更新）

**RunningHub 提供了官方的 `/openapi/v2/resource/list` 接口**，返回平台上所有公共模型的完整元数据，包括 `baseModel`（架构族）字段。这不是第三方 API，是 RunningHub 自己的数据。

```python
# 后端缓存调用（每日刷新一次即可）
POST https://www.runninghub.cn/openapi/v2/resource/list
→ 返回 1135+ 个模型，含 resourceType / nodeModelName / baseModel
```

#### 核心字段对应关系

```
工作流 JSON:
  UNETLoader.unet_name = "realDream_klein9BV1.safetensors"
                            ↕ 完全匹配
API 响应:
  resourceType   = "UNET"          ← 节点类型
  nodeModelName  = "realDream_klein9BV1.safetensors"
  baseModel      = "Flux2-Klein-9B" ← 架构族！
```

#### 量化信息

API **暂无明确量化字段**，量化从文件名关键词推断（`fp8_e4m3fn` / `int8` / `gguf` / `nf4`），准确率约 80%。`baseModelSubtype` 字段目前为空，可能是预留字段。

#### 优点
- **数据权威**：RunningHub 自己的模型库，100% 覆盖平台上的模型
- `nodeModelName` 与工作流 fieldValue 直接匹配，无需模糊搜索
- `baseModel` 字段直接给出架构族，无需正则推断
- `triggerWords` 可自动注入提示词
- `tags` 可用于模型筛选 UI

#### 缺点
- 需要缓存（1135 个模型 / 114 页，全量拉取约 114 次请求）
- 用户私有/上传模型不在公共列表中

#### 缓存策略

```python
# backend/data/runninghub/model_cache.json
# 每日凌晨或手动触发刷新
{
  "fetchedAt": "2026-05-28T00:00:00Z",
  "byFileName": {
    "realDream_klein9BV1.safetensors": {
      "resourceType": "UNET",
      "baseModel": "Flux2-Klein-9B",
      "resourceName": "realDream_Klein",
      "tags": ["摄影", "写实"]
    }
  }
}
```

## 四、推荐组合方案（更新后）

### 分阶段实施

```
Phase 1 — 低成本立即可做
├── 方案 C — 依赖图分析 → BFS 识别"模型联动组"（复用现有 BFS 引擎）
│   保存工作流时自动计算，标注哪些模型节点需联动替换
└── 方案 D (精简版) — 调用 /resource/list 缓存基础元数据
    只存 nodeModelName → baseModel 映射
    工作流展开时自动为模型节点标注架构标签（Flux2-Klein / SDXL / SD15）
    跨架构族修改时弹出警告

Phase 2 — 模型选择器（完整 UX）
└── 方案 D (完整版) + 方案 A 预设包
    点击模型字段 → 弹出下拉选择器，按 baseModel 过滤同族模型
    选择预设包 → 一键原子替换所有关联模型节点
```

### UI 设想（Phase 1 完成后）

```
┌─ 模型配置 (3) ────────────────────────────────────────┐
│  [Flux2-Klein] UNETLoader #4                           │
│    unet_name: [realDream_klein9BV1.safetensors ▼]      │
│                                                        │
│  [Flux2-Klein] CLIPLoader #11                          │
│    clip_name1: [t5xxl_fp8_e4m3fn.safetensors]          │
│    clip_name2: [clip_l.safetensors]                    │
│                                                        │
│  ✅ 同组 3 个节点架构一致 (Flux2-Klein)                │
└────────────────────────────────────────────────────────┘

修改 UNETLoader 为 SD1.5 模型后：

  ⚠️ 架构不一致！
  UNETLoader #4: SD 1.5
  CLIPLoader #11: Flux2-Klein  ← 不兼容
  建议同步更换 CLIPLoader 为 SD1.5 对应的 CLIP 模型
```

## 五、技术实现要点（更新后）

### 后端新增（`runninghub_client.py` + 新路由）

| 新函数 / 端点 | 职责 |
|--------|------|
| `fetch_model_cache() → dict` | 分页拉取 `/resource/list`，构建 `nodeModelName → metadata` 索引 |
| `find_model_groups(raw_nodes) → list` | BFS 找联动模型组（复用现有引擎） |
| `annotate_model_params(params, cache) → list` | 为参数列表中的模型字段附加 `baseModel` / `resourceType` |
| `GET /rh/model-cache/refresh` | 手动触发全量刷新（管理员用） |
| `GET /rh/model-cache/status` | 返回缓存状态（条数 / 最后刷新时间） |

### 前端新增

| 位置 | 变更 |
|------|------|
| `runninghub.ts` | 新增 `ModelMeta` 类型；`RHWorkflowParam` 新增 `modelMeta?` 可选字段 |
| `ParamGroup.vue` | 模型节点显示 `baseModel` 架构标签徽章 |
| `WorkflowCard.vue` | 检测到跨族修改时显示兼容性警告 |

### 缓存文件路径

```
backend/data/runninghub/model_cache.json  ← 全量索引（byFileName）
backend/data/runninghub/model_groups/     ← 按工作流缓存的模型分组
  {workflowId}_model_groups.json
```

## 六、结论（更新后）

| | 实现成本 | 用户体验 | 准确度 | 推荐阶段 |
|---|---------|---------|--------|------|
| 方案 A 预设包 | 中（需 UI + 存储） | ⭐⭐⭐⭐⭐ | 100% | Phase 2 叠加 |
| 方案 B 文件名推断 | 低（纯正则） | ⭐⭐⭐ | 70% | **已被方案 D 替代** |
| 方案 C 依赖图分组 | 低（复用 BFS） | ⭐⭐⭐⭐ | 100% | **Phase 1 优先** |
| 方案 D RH 官方 API | 中（缓存 + 索引） | ⭐⭐⭐⭐⭐ | 98%（公共模型）| **Phase 1 核心** |

**最终推荐路径**：
- C + D 并联实施（Phase 1）：BFS 找模型组 + 官方 API 给出 baseModel 标签 → 兼容性警告
- A 叠加（Phase 2）：预设包一键切换，配合 D 的模型选择器 UI
