# 工作流输入端语义分析

> 日期: 2026-05-28
> 状态: 分析完成，待决策

---

## 背景

ComfyUI 工作流通过 RunningHub 云端执行时，系统需解析 JSON 中的节点参数并暴露给用户编辑。当前 `NODE_TYPE_MAP`（`backend/core/runninghub_client.py`）对所有 `LoadImage` 节点统一标记为"输入图片"。

**问题：** 复杂工作流（如多视图角色生成）包含 15+ 个 `LoadImage` 节点，用户看到 15 个相同的"输入图片"标签，无法区分每个输入口的用途——鸡同鸭讲。

---

## 样本工作流分析

### 样本 A：动漫反推流 V2.15（API 格式，858 行）

简单的**图生图 + 反推标签**工作流。

| 节点 ID | class_type | 字段 | 语义 |
|---------|-----------|------|------|
| 188 | LoadImage | image | **唯一图片输入** — 喂给 WD14 Tagger + VLM + VAEEncode |

其他可调输入：

| 类型 | 节点示例 | 普遍性 |
|------|---------|--------|
| 提示词 | CLIPTextEncode (11, 12), CR Prompt Text (190, 366) | ✅ 通用 |
| 种子 | Seed (rgthree) (91, 460) | ✅ 通用 |
| 分辨率 | TTResolutionSelector (575), EmptyLatentImage (28) | ✅ 通用 |
| 去噪强度 | easy float (207) = 0.7 | ✅ 通用 |
| 布尔开关 | PrimitiveBoolean (365) = txt2img/img2img 切换 | ⚠️ 工作流特定 |
| LoRA | LoraLoaderModelOnly (582) | ⚠️ 半通用 |

**结论：1 个图片输入，语义明确，无歧义。**

### 样本 B：Flux2+Klein 多视图工作流（UI 格式，27905 行）

复杂的**角色多视图生成**工作流。

#### LoadImage 节点统计：15 个

按 Group 分区归属：

| Group 名称 | 含义 | 内含 LoadImage 数 | 用户是否需要输入 |
|-----------|------|-------------------|----------------|
| Input image | 主角色参考图 | 1 (`input image0`) | ✅ **是 — 唯一必填** |
| Pose image | 各角度骨骼参考 | 4（正/侧/背/45°） | ❌ 预设，不改 |
| face | 脸部一致性参考 | 2 | ❌ 通常自动从输入派生 |
| Front / Rear / side up / 45° / Front up | 各视角采样区 | 多个 | ❌ 内部处理 |

#### 控制开关（Groups 标题暴露）

| 开关 | 类型 | 默认 |
|------|------|------|
| ←开/关输入图参考 | Boolean | 开 |
| ↑骨骼图开关 | Boolean | — |
| ↑脸部参考开关 | Boolean | — |
| 切换↑输入图/正视图参考 | Boolean | — |
| 开启/关闭↑脸部增强 | Boolean | — |
| 自定义提示词↑切换 | Boolean | — |

#### 节点命名线索

工作流使用 `easy setNode` / `easy getNode` 传递数据，命名含语义：
- `input image0` — 主输入图（被 7 个 getNode 引用）
- `input image1` — 第二输入（脸部参考等）
- `output image1` — 最终输出

**结论：用户只需输入 1 张图，但系统暴露了 15 个 LoadImage 节点，完全无法区分。**

## 输入类型普遍性总结

| 输入类型 | class_type 示例 | 普遍性 | 备注 |
|---------|----------------|--------|------|
| 提示词 | CLIPTextEncode, CR Prompt Text | ✅ 通用 | 所有生图工作流都有 |
| 种子 | KSampler.seed, Seed (rgthree) | ✅ 通用 | |
| 采样步数 | KSampler.steps | ✅ 通用 | |
| CFG | KSampler.cfg | ✅ 通用 | |
| 去噪 | KSampler.denoise | ✅ 通用 | |
| 分辨率 | EmptyLatentImage, TTResolutionSelector | ✅ 通用 | |
| **图片输入** | **LoadImage** | **❌ 不通用** | **同为 LoadImage，语义角色完全不同** |
| 模型/LoRA | UNETLoader, LoraLoader | ⚠️ 半通用 | 高级用户可能要换 |
| 布尔开关 | PrimitiveBoolean | ⚠️ 工作流特定 | 含义取决于下游连接 |

**关键结论：图片输入是唯一不具有普遍性的关键输入类型。**

## 解决方案候选

### 方案 1：UI 格式元数据自动推断

UI 格式 JSON 包含 API 格式没有的上下文信息：

| 元数据 | 来源 | 示例 |
|--------|------|------|
| **groups** | `json.groups[]` | `{title: "Input image", bounding: [...]}` |
| **node title** | `node.title` | `"➡️input image0"` |
| **setNode/getNode 名称** | `widgets_values[0]` | `"input image0"` |
| **node 位置** | `node.pos` | 可判断所属 group |

推断流程：
1. 解析 groups 的 bounding box
2. 每个 LoadImage 节点按 pos 匹配所属 group
3. 用 group title 作为语义标签

**优点：** 自动化，无需人工  
**缺点：** 仅 UI 格式可用；RunningHub API `getJsonApiFormat` 返回的是 API 格式（无 group/pos 信息）；推断可能不准

### 方案 2：下游连接启发式

根据 LoadImage 输出连接的下游节点类型推断用途：

| 下游节点 | 推断角色 |
|---------|---------|
| IPAdapter / StyleAligned | 风格/角色参考图 |
| ControlNet (OpenPose/Canny/Depth) | pose/边缘/深度图 |
| VAEEncode | img2img 源图 |
| FaceSwap / ReActor / InstantID / FaceDetailer | 脸部参考 |
| WD14Tagger / VLM / llama_cpp | 打标分析用图 |
| ImageComposite / ImageConcat | 拼接素材 |

**优点：** 在 API 格式中也可用（API 格式保留了连接信息 `[nodeId, slotIndex]`）  
**缺点：** 规则需不断扩充；某些中间节点（如 Any Switch）会打断推断链

### 方案 3：保存时人工标注（推荐配合方案 2）

用户保存工作流时，系统先用启发式预填，用户再确认/修正每个图片输入的语义角色。

#### 语义角色枚举

```
reference    — 主参考图（用户必填，高优先级）
pose         — 骨骼/姿势引导图（预设，一般不改）
face         — 脸部参考（可选）
style        — 风格参考（可选）
mask         — 遮罩（可选）
background   — 背景图（可选）
internal     — 内部节点，不暴露给用户
```

#### 存储

在保存的工作流 JSON 中扩展 `imageInputs` 字段：

```json
{
  "id": "workflow_xxx",
  "name": "多视图工作流",
  "imageInputs": [
    {
      "nodeId": "671",
      "fieldName": "image",
      "role": "reference",
      "label": "角色参考图",
      "required": true
    },
    {
      "nodeId": "149",
      "fieldName": "image",
      "role": "pose",
      "label": "正面骨骼图",
      "required": false
    },
    {
      "nodeId": "728",
      "fieldName": "image",
      "role": "internal",
      "label": "",
      "required": false
    }
  ]
}
```

#### 前端展示

- `role=internal` 的节点在参数编辑器中**隐藏**
- `role=reference` 显示在最上方，标记为必填
- 其他角色按 role 分组显示，附带角色说明

**优点：** 精确；一次标注、长期受益  
**缺点：** 需要用户理解工作流结构（首次成本）

---

## 推荐方案：方案 2 + 3 组合

| 步骤 | 动作 |
|------|------|
| 1. 保存工作流时 | 后端解析 API 格式 JSON，用下游连接启发式预填每个 LoadImage 的 role |
| 2. 前端确认 | 弹出标注面板，用户确认/修正每个图片输入的角色和标签 |
| 3. 持久化 | `imageInputs` 存入工作流 JSON |
| 4. 运行时 | 参数编辑器按 role 分组展示，`internal` 隐藏 |

---

## 输出端

输出端相对简单，不需要语义标注：

- `SaveImage` / `SaveAnimatedWEBP` / `SaveVideo` — 直接从任务结果 outputs 拿到媒体 URL
- RunningHub API `outputs` 返回 `fileUrl` 列表
- 前端已有 TaskRunner 组件展示输出预览
