## 候选项 1 — `core/runninghub_client.py` 过度膨胀

### 涉及文件
`backend/core/runninghub_client.py`（713 行，硬限 400 行，**超标 78%**）

### 问题

文件名叫 "client"，实际承担了 5 个独立职责：

| 职责 | 行区间 | 应独立为 |
|------|--------|---------|
| HTTP 客户端基础设施 | 105–140 | 保留在 `runninghub_client.py` |
| API Key / 代理配置读取 | 105–130 | 同上 |
| 规则引擎：NODE_TYPE_MAP（22节点）+ 解析逻辑 | 22–280 | `workflow_analyzer.py` |
| 图片语义推断（BFS + _DOWNSTREAM_ROLE_MAP 25项） | 286–423 | `image_input_inferrer.py` |
| 参数功能分组（_GROUP_DEFINITIONS 6组 + BFS） | 426–503 | 可合并入 `workflow_analyzer.py` |
| 模型联动组分析（BFS 反向图）| 612–677 | 同上 |
| 参数元数据注入 | 680–690 | 同上 |

**删除测试**：如果删掉这个文件，复杂性不会消失，它会分裂到 3 个调用方中 → 说明模块值得存在，但需要**水平拆分为更深的子模块**。

### 方案

```
core/
  runninghub_client.py    ← 保留：HTTP 通信（test/fetch/create_task/poll/cancel/upload）
  workflow_analyzer.py    ← 新建：NODE_TYPE_MAP + analyze_nodes_by_rules
                                  + group_params_by_function + 模型联动组 + annotate_meta
  image_inferrer.py       ← 新建：_DOWNSTREAM_ROLE_MAP + infer_image_inputs + BFS
```

拆分后：
- `runninghub_client.py`：~200 行（只剩 HTTP 逻辑）
- `workflow_analyzer.py`：~280 行（分析规则全集中）
- `image_inferrer.py`：~140 行（图片语义推断）

### 收益

- **局部性**：改 NODE_TYPE_MAP 只在 `workflow_analyzer.py`，不触 HTTP 代码
- **杠杆**：`workflow_analyzer` 可单独测试，无需 mock HTTP 客户端
- **行数**：三文件均低于 400 行硬限

### 推荐强度：**强烈推荐**

## 候选项 2 — `routers/runninghub.py` 路由层业务逻辑溢出

### 涉及文件
`backend/routers/runninghub.py`（402 行，**恰好触碰硬限**）

### 问题

路由层承担了两类不应属于它的职责：

**A. 文件系统操作直接在路由层**

`_DATA_DIR`, `_ensure_data_dir`, `json.loads`, `filepath.write_text` 散落在5个端点函数体内：
```python
# list_saved_workflows: 路由层直接遍历 glob, 手动拼 dict
for f in sorted(_DATA_DIR.glob("*.json")):
    data = json.loads(f.read_text(...))
    items.append({ "workflowId": ..., "name": ..., ... })   # 17 行投影逻辑

# save_workflow: 路由层直接读写 JSON + 调用 5 个分析函数
analyzed = rh.analyze_nodes_by_rules(body.raw_nodes)
existing["analyzedParams"] = analyzed
...
rh.annotate_params_with_meta(analyzed, by_filename)
filepath.write_text(...)
```

**B. 配置写入路径与读取路径不对称**

- 读：`get_config()` → `core/config.py`
- 写：路由层直接 `yaml.safe_load` / `yaml.dump` 操作文件，绕过 config 层

**C. `_refresh_task` 全局 Task 放在路由模块**

`_refresh_task: asyncio.Task | None = None` 是 model_cache 的内部状态，却住在路由文件里。

### 方案

```
services/
  workflow_store.py       ← 新建：工作流 JSON CRUD
                              list_workflows() / get_workflow(id) / save_workflow(...)
                              delete_workflow(id) / patch_model_overrides(id, [...])
  rh_config_service.py    ← 新建（或扩展 core/config.py）：
                              get_rh_config() / save_rh_config(...)
```

路由层退化为薄路由：每个端点 ≤ 5 行，只做参数校验 + service 调用 + 响应格式化。

`_refresh_task` 迁移到 `model_cache_service.py`，由 `start_refresh(api_key, proxy)` 管理。

### 收益

- **局部性**：工作流 CRUD 变更只在 `workflow_store.py`
- **杠杆**：`list_workflows()` 的字段映射逻辑集中一处，ModelChecker/WorkflowManager 都受益
- **行数**：路由文件可缩减至 ~150 行

### 推荐强度：**强烈推荐**

```
frontend/src/components/workflow/ParamGroup.vue:18-25
frontend/src/components/workflow/ModelGroupItem.vue:23-32
frontend/src/components/workflow/ModelSelector.vue:18-25
```

### 问题

同一个架构色彩映射字典**逐字复制了三遍**（略有差异，`ModelGroupItem` 版本键略有不同，已产生内容漂移）：

```ts

'Flux2-Klein': 'bg-violet-600/20 text-violet-300 border-violet-500/40',

```

同族模型已出现视觉不一致，随着架构种类增加会越来越难同步。

```ts
  'Flux2-Klein': '...',
  'Flux1':       '...',
  ...

  return ARCH_COLORS[baseModel] ?? 'bg-gray-600/20 text-gray-400 border-gray-600/40'
}
```

三处组件改为 `import { archBadgeClass } from '@/utils/modelArchColors'`。

### 收益

- **局部性**：新增 Wan2.2、HiDream-Fast 等架构只改一处
- **一致性**：三个组件视觉保证同源

### 推荐强度：**强烈推荐**（改动最小，收益最直接）

## 候选项 5 — `ParamGroup.vue` 过度复杂

### 涉及文件
`frontend/src/components/workflow/ParamGroup.vue`（256 行，建议 150 行，**超标 70%**）

### 问题

7 个计算逻辑全塞在同一 SFC：

```
modelSortRank()      — 节点排序规则
fieldSortRank()      — 字段排序规则
nodeClass()          — 节点标题解析（正则剥离"节点44(ClassName)"格式）
nodeGroups           — 按节点合并 + 双重排序（computed）
groupBaseModel       — UNET 权威架构族推断（computed）
isIncompatible()     — 兼容性判断（依赖 groupBaseModel）
fieldFullWidth()     — 布局决策函数
```

其中**兼容性判断逻辑**（`groupBaseModel` + `isIncompatible`）同时在 `ModelChecker.vue:72-95` 中重新实现了一套不同的版本（基于多数投票）— 两套逻辑目标相同但算法不同。

### 方案

```ts
// composables/useModelCompat.ts（新建）
export function useModelCompat(params: Ref<RHWorkflowParam[]>) {
  const dominantBaseModel = computed(...)  // 统一投票算法
  const compatStatus = computed(...)       // per-param 兼容性
  return { dominantBaseModel, compatStatus }
}
```

`ParamGroup.vue` 和 `ModelChecker.vue` 统一调用此 composable，消除两套并行逻辑。

`ParamGroup.vue` 模板中的节点分组排序提取为 `useParamGroupSorter.ts`。

### 收益

- **局部性**：兼容性算法调整只改 composable
- **一致性**：工作流编辑器和模型检测器使用相同判断逻辑
- **行数**：`ParamGroup.vue` 可缩减至 ~130 行

### 推荐强度：**值得探索**

## 候选项 6 — 模型字段关键词前后端双维护

### 涉及文件

```python
# backend/core/runninghub_client.py:624
_MODEL_FIELD_KEYWORDS: tuple[str, ...] = ("name", "path", "ckpt")
```

```ts
// frontend/src/components/workflow/ModelChecker.vue:41
function isModelParam(p: RHWorkflowParam) {
  return !!p.modelMeta || /name|path|ckpt/i.test(p.fieldName)
}
```

### 问题

判断"是否为模型类型字段"的逻辑在前后端各维护一份，关键词完全相同。如果后端新增 `weight` 或 `model_file` 类关键词，前端不会同步，导致 `ModelChecker.vue` 漏显示字段。

**本质**：后端已通过 `annotate_params_with_meta` 给模型字段打上 `modelMeta` 标记。前端的正则是对这个标记失效时的后备逻辑，但这个后备路径已被 `ModelChecker` 依赖为主路径。

### 方案

A. **强化后端标注**：`annotate_params_with_meta` 即使模型不在缓存里也打上 `isModelField: true` 标记，前端只依赖该字段。  
B. 或：将 `isModelParam` 封装为 `utils/paramUtils.ts` 的公共函数，同时废弃后端的正则逻辑，改为全部由前端 `modelMeta` 字段驱动。

方案 A 更彻底（减少前端需要了解的后端规则）。

### 收益

- **杠杆**：新增字段类型只需改后端的 `_MODEL_FIELD_KEYWORDS`，前端自动生效

### 推荐强度：**值得探索**

## 候选项 7 — 路由 URL 命名风格不一致

### 涉及文件
`backend/routers/runninghub.py`

### 问题

同一路由文件中混用单数/复数、路径风格：

| 端点 | URL | 问题 |
|------|-----|------|
| 列表 | `GET /workflows` | 复数 ✅ |
| 详情 | `GET /workflow/{id}` | **单数** ❌ |
| 保存 | `POST /workflow/save` | 动词 + 单数 ❌ |
| 拉取 | `POST /workflow/fetch` | 动词 + 单数 ❌ |
| 分析 | `POST /workflow/analyze` | 动词 + 单数 ❌ |
| 模型替换 | `PATCH /workflows/{id}/models` | 复数 ✅ |

RESTful 规范：名词复数 + HTTP 语义动词，不在 URL 里放动词。

### 方案

```
GET    /workflows              → 列表
POST   /workflows              → 创建/更新（save 语义）
GET    /workflows/{id}         → 详情
DELETE /workflows/{id}         → 删除
PATCH  /workflows/{id}/models  → 模型替换

POST   /workflows/fetch        → 远端拉取（非 CRUD 操作，可保留动词）
POST   /workflows/analyze      → 分析（同上）
```

### 收益

- **接口一致性**：前端 `runninghub.ts` 里 URL 字符串不再需要区分 `/workflow` vs `/workflows`

### 推荐强度：**试探性**（breaking change，需前后端同步修改）

## 候选项 9 — `model_cache_service.py` 全局可变状态不安全

### 涉及文件
`backend/services/model_cache_service.py`（218 行，目前合规）

### 问题

两个模块级可变状态在多线程（`asyncio.to_thread` 已在线程池中运行同步代码）下有竞态风险：

```python
_cache_mem: dict[str, Any] = {}          # 写：load_cache / fetch_and_save
_refresh_progress: dict[str, Any] = {}   # 写：fetch_and_save
```

`_refresh_progress` 在 `fetch_and_save` 内循环中被替换（`= {}`，非原地更新），路由层的 `_refresh_task` 完成后不会重置 progress，下次打开设置页还会显示上次的进度快照。

此外，`fetch_and_save` 函数体 **47 行**，超过单函数 40 行硬限（实测含空行注释约 37 非空行，但含注释算 47）。

### 方案

用 `dataclass` 封装状态，避免裸 dict 替换：

```python
@dataclass
class _RefreshProgress:
    page: int = 0
    total_pages: int | None = None
    count: int = 0

_progress = _RefreshProgress()
```

`fetch_and_save` 完成后调用 `_progress = _RefreshProgress()` 重置。

`_cache_mem` 增加简单的文件修改时间检测，避免进程内多实例（如测试）读到脏数据。

`fetch_and_save` 的循环体拆出 `_process_page_data(data, by_filename)` 辅助函数，使函数体 ≤ 40 行。

### 收益

- **局部性**：进度状态变更集中在 service 内，路由层不再持有 `_refresh_task`
- **正确性**：消除刷新完成后进度残留

### 推荐强度：**试探性**（当前未造成 bug，但为将来并发铺路）
